# NIST SP 800-115 — Technical Guide to Information Security Testing and Assessment

> **Uso autorizado e responsável.** O SP 800-115 enfatiza planejamento, regras de engajamento e coordenação prévia. Nenhuma técnica intrusiva deve ser executada sem **escopo aprovado, RoE e autorização por escrito**. Documente cada ação para auditoria.

O **NIST SP 800-115** é o guia técnico do NIST para avaliações de segurança. Diferente do PTES (foco em pentest ofensivo) e do WSTG (foco em web), ele cobre o espectro completo de **assessment** e estrutura o trabalho em uma metodologia de 4 fases com três famílias de técnicas.

## Famílias de técnicas
1. **Review Techniques** (examine) — análise passiva: documentação, logs, regras, configs.
2. **Target Identification & Analysis** (identify) — descoberta de hosts/serviços/vulns.
3. **Target Vulnerability Validation** (exploit) — confirmar vulnerabilidades explorando-as.

## As 4 fases da metodologia
1. **Planning** — objetivos, escopo, RoE, regras, logística.
2. **Discovery** — coleta de informação, varredura e análise de vulnerabilidades.
3. **Attack** — ganho de acesso, escalonamento, system browsing, instalação de ferramentas (loop com Discovery).
4. **Reporting** — documentação, achados, mitigação.

```
  Planning ──▶ Discovery ⇄ Attack ──▶ Reporting
                 ▲___________│  (loop: novos hosts/info realimentam Discovery)
```

---

## Fase 1 — Planning
Define a fundação legal e operacional do assessment.

- [ ] Objetivos do assessment e critérios de sucesso.
- [ ] Escopo (sistemas, redes, aplicações, exclusões explícitas).
- [ ] **Rules of Engagement (RoE)** documentadas e assinadas.
- [ ] Autorização formal da gestão/proprietário do ativo.
- [ ] Logística: janelas, contatos, escalonamento, handling de dados.
- [ ] Seleção de técnicas (review / identification / validation) por tipo de ativo.
- [ ] Definição de papéis: assessor, ponto focal, observadores do blue team.

**Entrega:** Assessment Plan + RoE assinados.

---

## Fase 2 — Discovery
Duas atividades: (a) coleta de informação e (b) análise de vulnerabilidades.

### Review Techniques (examine — passivo)
- [ ] **Documentation review** (políticas, arquiteturas, baselines).
- [ ] **Log review** (autenticação, firewall, IDS) para detectar desvios.
- [ ] **Ruleset review** (firewall/router/ACL) e **system config review** (hardening).
- [ ] **Network sniffing** (em rede autorizada) para inventário e exposição.
- [ ] **File integrity checking** (baselines de hashes).
```bash
# Captura passiva (em rede autorizada)
tcpdump -i eth0 -nn -w captura.pcap
# Conferência de integridade
sha256sum -c baseline.sha256
```

### Target Identification & Analysis (identify — ativo)
- [ ] **Network discovery** (hosts vivos, topologia).
- [ ] **Port & service identification** (portas, versões, banners).
- [ ] **Vulnerability scanning** (validar aplicabilidade, reduzir FP).
- [ ] **Wireless scanning** (SSIDs, rogue APs) quando no escopo.
```bash
nmap -sn 10.0.0.0/24                       # network discovery
nmap -sV -sC -p- --min-rate 1500 -oA disc/alvo 10.0.0.10  # service ID
nuclei -l hosts.txt -severity high,critical               # vuln scan
airmon-ng / kismet                          # wireless (se no escopo)
```
**Saída:** inventário de alvos + lista priorizada de vulnerabilidades candidatas.

## Fase 3 — Attack (Target Vulnerability Validation)
Confirma vulnerabilidades **explorando-as de forma controlada**. O NIST descreve um ciclo: ganhar acesso → escalar privilégios → navegar no sistema → instalar ferramentas adicionais, realimentando a Discovery com novas informações.

- [ ] **Gaining access** — explorar vuln validada (PoC mínima, sem dano).
- [ ] **Privilege escalation** — local/domínio, documentando o caminho.
- [ ] **System browsing** — identificar dados/ativos alcançáveis (sem exfiltração real).
- [ ] **Install additional tools** — apenas se necessário e autorizado; registrar tudo.
- [ ] Realimentar a fase de **Discovery** com hosts/credenciais descobertos.
- [ ] **Cleanup**: remover artefatos, contas e ferramentas instaladas.

```bash
# Validação de vulnerabilidade (exemplo de framework, uso controlado)
msfconsole -q -x "use exploit/...; set RHOSTS alvo; set LHOST <ip-equipe>; run"

# Enumeração pós-acesso para escalonamento
linpeas.sh        # Linux
winpeas.exe       # Windows
```

> **Controle de risco:** o SP 800-115 recomenda avaliar o impacto potencial de cada técnica intrusiva (instabilidade, indisponibilidade) e coordenar com a equipe operacional. Evite exploits instáveis ou destrutivos sem aprovação explícita no RoE.

---

## Fase 4 — Reporting
- [ ] Metodologia, escopo, ferramentas e timeline do assessment.
- [ ] Achados com evidência, severidade e **causa raiz**.
- [ ] **Recomendações de mitigação** priorizadas por risco.
- [ ] Sumário executivo (postura de risco) + detalhe técnico.
- [ ] Limitações do teste (cobertura, restrições de janela/escopo).
- [ ] Plano de remediação e, quando aplicável, retest.

## Comparação rápida (quando usar cada um)

| Aspecto | NIST SP 800-115 | PTES | OWASP WSTG |
|---|---|---|---|
| Foco | Assessment técnico amplo (review+identify+exploit) | Pentest ofensivo end-to-end | Teste de aplicações web/API |
| Fases | 4 (Planning, Discovery, Attack, Reporting) | 7 fases | Categorias por domínio |
| Forte em | Governança, examine/review, conformidade | Threat modeling, exploração, pós-exploração | Profundidade técnica web/API |
| Uso típico | Programas de avaliação corporativa/compliance | Engajamentos de pentest | Camada web dentro de PTES/NIST |

## Remediação transversal (recomendações NIST)
- Hardening e baselines de configuração (revisar regras/configs periodicamente).
- Gestão de vulnerabilidades contínua (não pontual) e patching.
- Segmentação de rede, menor privilégio e MFA.
- Monitoração/log review como controle detectivo (validado pelo teste).
- Reavaliação periódica e após mudanças significativas.
